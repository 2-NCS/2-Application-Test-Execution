# 설계과제 — 애플리케이션 테스트 수행 (S-Library)

## 파트 A. 테스트 수행 (NCS 1)
- 단위 테스트는 `@ExtendWith(MockitoExtension.class)` 환경에서 Repository 를 `@Mock` 으로 대체하고 `@InjectMocks` 로 서비스를 주입하여 작성합니다.
- 다음 경계·예외 케이스를 포함합니다.
  - 정상 대출 : 재고가 1 줄고 반납기한(dueAt)이 설정됩니다.
  - 재고 0 경계 : 대출 가능 부수가 0 이면 대출이 차단됩니다.
  - 미존재 도서 : 존재하지 않는 도서를 대출하면 예외가 발생합니다.
  - 중복 반납 : 이미 반납된 대출을 다시 반납하면 차단되고 재고가 한 번만 증가합니다.
  - 연체 경계 : 반납기한 당일은 연체가 아니고, 기한을 지난 시점부터 연체입니다.
- 슬라이스 테스트는 `@DataJpaTest`(Repository)와 `@WebMvcTest`(Controller, `@MockitoBean`)로 작성합니다.
- 통합 테스트는 `@SpringBootTest` + `MockMvc` 로 대출 후 재고 반영을 확인합니다.
- 각 테스트의 입력·기대결과를 `테스트케이스_명세서.md` 에 명세하고, 최초 실행 결과(실패 포함)를 `테스트_실행결과서.md` 에 기록합니다.

## 파트 B. 결함 식별·기록 (NCS 2-1)
- 실패한 테스트가 가리키는 결함을 식별하여 `결함_관리대장.md` 에 기록합니다.
- 결함마다 유형(분류)·심각도(severity)·우선순위(priority)·재현 절차·기대결과/실제결과를 적습니다.
- 심각도(영향의 정도)와 우선순위(수정의 긴급도)는 서로 다른 속성이므로 구분하여 판단합니다.

## 파트 C. 결함 조치·회귀 (NCS 2-2 / 2-3)
- 결함의 원인을 분석하여 `src/main` 코드를 수정합니다.
- 수정 후 전체 테스트를 다시 실행하여 모두 통과함을 확인합니다(회귀 테스트로 부작용 없음 확인).
- 결함조치마다 Git 커밋을 남겨 변경 이력을 관리하고, `결함조치_결과서.md` 에 원인·조치·회귀 결과를 정리합니다.

## 제약
- 추가 라이브러리 없이 Spring Boot Test(JUnit5·Mockito·MockMvc)와 인메모리 H2 만 사용합니다.
- 테스트 클래스/메서드 시그니처는 제공된 골격을 따르고, TODO 위치를 채웁니다.
- 결함 수정은 동작을 바로잡는 최소 범위로 하고, 통과를 위해 테스트를 약화시키지 않습니다.

> 참고 표준 : 테스트 문서·결함 보고는 ISO/IEC/IEEE 29119-3 의 테스트 케이스 명세·결함(Incident) 보고 템플릿, 결함 보고 필드·심각도/우선순위 구분은 ISTQB 기준을 따른다. (결함 진단·조치형 과제는 NCS 권장 평가방법인 '문제해결 시나리오'에 해당)
