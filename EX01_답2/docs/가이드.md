# 가이드 — 막힐 때 참고

## 테스트 작성 힌트
| 상황 | 사용 도구 |
|:--|:--|
| 서비스 로직만 빠르게 검증 | JUnit5 + Mockito (`@Mock`, `@InjectMocks`, `when(...).thenReturn(...)`) |
| 예외 발생 검증 | `assertThrows(BizException.class, () -> ...)` |
| Repository(JPA) 쿼리 검증 | `@DataJpaTest` (인메모리 H2 자동 구성) |
| 컨트롤러 응답/상태 검증 | `@WebMvcTest` + `MockMvc` + `@MockitoBean`(서비스 대체) |
| 전체 흐름 검증 | `@SpringBootTest` + `@AutoConfigureMockMvc` |

## 핵심 힌트
- Mockito 단위 테스트에서는 Repository 가 반환할 값을 `when(...).thenReturn(...)` 으로 정해 두고, 서비스의 계산·분기만 검증합니다.
- 경계값은 "직전 / 같은 값 / 직후" 세 지점을 함께 확인하면 결함이 잘 드러납니다(예: 재고 0, 반납기한 당일).
- `@WebMvcTest` 는 컨트롤러 계층만 띄우므로 서비스는 `@MockitoBean` 으로 대체하고, 검증 실패(400)는 잘못된 요청 본문을 보내 확인합니다.
- `@SpringBootTest` 는 전체 컨텍스트를 띄우므로 인메모리 H2 에 직접 저장한 뒤 API 를 호출해 결과를 확인합니다.
- Spring Boot 3.4 이상에서는 `@MockBean` 이 아니라 `@MockitoBean` 을 사용합니다.

## 결함 조치 절차
1. 테스트를 먼저 완성하고 실행합니다. 일부 테스트가 실패하면 그 실패가 결함의 단서입니다.
2. 실패 내용을 `결함_관리대장.md` 에 기록합니다(유형·심각도·우선순위·재현 절차·기대결과/실제결과).
3. 원인을 분석하여 `src/main` 의 해당 로직을 수정합니다.
4. 전체 테스트를 다시 실행하여 모두 통과하는지 확인합니다(회귀 테스트로 다른 기능이 깨지지 않았는지 확인).
5. 결함 조치마다 Git 커밋을 남기고, `결함조치_결과서.md` 에 원인·조치·회귀 결과를 정리합니다.

## 실행 시 참고
- 프로젝트가 한글 경로에 있으면 `gradlew test`(명령줄)에서 테스트 클래스 로딩 오류가 날 수 있습니다. 이때는 IntelliJ 의 테스트 실행기로 돌리거나 영문 경로에서 실행합니다.
